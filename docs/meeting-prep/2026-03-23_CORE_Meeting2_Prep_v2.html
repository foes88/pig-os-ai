04
기술 아키텍처
FastAPI + PostgreSQL + TimescaleDB. 완전 신규 설계.
4.1 백엔드: FastAPI(Python)
권고: FastAPI(Python) 선택. AI 분석 기능을 만들려면 Python이 필수. 다른 기술로 시작하면 나중에 Python을 따로 붙여야 해서 이중 작업이 된다.
Java 전환: Python은 빠르게 만들기 좋고, Java는 대규모로 버티기 좋다. 지금은 빠르게 만드는 게 먼저. 농장 2,000개 넘거나 대형 고객이 SLA 99.9%를 요구할 때 전환 시작.
4.2 DB 설계 — 완전 신규
Key Message
CORE는 기존 시스템(피그플랜 Oracle DB)과 완전히 독립된 신규 설계로 진행한다.
레거시에 얽매이지 않고, 글로벌 양돈 표준에 맞춰 처음부터 설계하는 것이 더 빠르고 깨끗하다.
원칙내용이유
멀티테넌시
(Schema-per-tenant)		하나의 시스템에서 여러 농장을 운영하되,
각 농장의 데이터는 완전히 분리된 공간(스키마)에 저장		A농장 데이터가 B농장에 절대 보이지 않음.
규정 준수, 농장 단위 백업 가능
5개 시장 동시 대응country_configs로 지역별 설정단일 스키마로 전 시장 커버
모듈형 구조11개 모듈 독립 배포Phase별 점진 확장
오프라인 동기화sync_queue + LWW인터넷 없는 농장 대응
감사 추적모든 변경 → audit_log규정 준수 + 데이터 신뢰성
Bottom Line FastAPI + PostgreSQL (Schema-per-tenant) + TimescaleDB(IoT). 완전 신규 설계, 바로 개발 착수 가능.

5.3 KPI 계산 공식

NPD = SUM(이유~재교배 간격) [12개월]
FCR = SUM(사료소비 kg) / SUM(체중증가 kg)
분만율 = 분만 모돈수 / 교배 모돈수 × 100
