## 2. 미국 양돈 프로그램 가격 & 중점 관리 포인트

> **핵심 메시지:** 경쟁사는 가격을 안 보여준다. 그게 우리의 기회다.
> 그러나 "무료"만으로는 안 된다 — 그들이 왜 돈을 받는지 이해해야 한다.

### 2.1 미국 시장 주요 경쟁사 가격 현황

| 소프트웨어 | 확인된 가격 | 과금 방식 | 출처 신뢰도 |
|----------|-----------|---------|-----------|
| **PigKnows** (Valstone) | **비공개** — Custom Quote | 모돈 수 기반 추정 | 공식 미확인 |
| **PigCHAMP** | **비공개** — 무료 체험 후 영업 | Per Sow/년 추정 | 공식 미확인 |
| **MetaFarms** | **$0.05~$0.25/출하돈** + setup $0.30~$1.00/sow | 출하 수 기반 | 2018년 기사 — 현행 미확인 |
| **Cloudfarms** | **$500/년** 시작 (리뷰 사이트) | 모듈형 추가 과금 | SourceForge — 벤더 미확인 |
| **MTech Systems** | **비공개** — Enterprise만 | Integrator 연간 계약 | 영업 문의 필요 |

**⚠️ 위 가격은 모두 추정값. 실제 영업 문의 없이는 확인 불가.**
**→ Action: MetaFarms, PigKnows에 "소규모 농장" 가장해서 견적 문의 권장**

### 2.2 미국 농장이 돈을 쓰는 이유 (중점 관리 포인트)

미국 Integrator들이 양돈 SW에 기꺼이 돈을 쓰는 이유:

| 관리 포인트 | 설명 | SW 없을 때 손실 |
|-----------|------|--------------|
| **PSY 1두 개선** | 모돈 1두당 연 1마리 더 = 미국 평균 농장(1000모돈) 연 $50,000+ 추가 수익 | ROI 계산 즉시 가능 |
| **Feed Mill 발주 자동화** | 수작업 사료 발주 오류 = 손실. 대형 농장 사료비 연 수억원 | 오발주 1건 = SW 1년치 비용 |
| **질병 조기 탐지** | PRRS·PED·ASF 조기 경보 = 폐사율 차이 | 전염병 1회 = 수천만원 손실 |
| **Pork Checkoff 규정 준수** | PQA Plus 인증 없으면 대형 유통사 계약 불가 | 계약 미체결 = 판로 없음 |
| **도체 등급 피드백** | 도축 성적을 역으로 교배 결정에 반영 | 데이터 없으면 개선 불가 |
| **인력 효율화** | 현장 작업자 1명이 모바일로 데이터 입력 | 데이터 담당 추가 인건비 절감 |


### 2.3 해외 경쟁사 가격 대응 전략

> **대표님 보고 핵심:** "우리는 왜 무료인가?" → "무료가 전략이다. 단, 조건이 있다."

```
경쟁사가 $500~$3,000/년을 받는 이유:
  → 데이터 마이그레이션 비용 (전환 비용 높아야 Lock-in)
  → 영업 인력 운영비 포함
  → 27년 된 브랜드 프리미엄

우리가 무료로 줄 수 있는 이유:
  → 피그플랜 27년 개발 비용 이미 회수
  → 글로벌 데이터 자산(Pigsignal) 확보가 진짜 목적
  → Phase 1에서 수익보다 시장 침투가 먼저
```

**대응 포지셔닝 3줄:**
1. **"경쟁사가 $500 받을 때 우리는 0원"** → SEA 시장 즉시 진입
2. **"경쟁사가 API 없을 때 우리는 오픈"** → ESF·도축장·ERP 자유 연동
3. **"경쟁사가 아시아어 없을 때 우리는 한/베/태"** → 현지 농장 즉시 사용

## 3. 기술 스택 결정 사항

> **핵심 메시지:** 기술 선택은 "지금 빠르게"와 "나중에 버티게" 사이의 균형이다.
> 잘못 고르면 6개월 후 전부 다시 짜야 한다.

### 3.1 백엔드 선택: FastAPI vs Node.js

| 기준 | FastAPI (Python) | Node.js (TypeScript) |
|------|-----------------|---------------------|
| **개발 속도** | ✅ 빠름 — Python 팀 역량 활용 | ✅ 빠름 — JS 생태계 |
| **AI/ML 연동** | ✅ 최강 — NumPy, pandas, 모델 직접 호출 | ⚠️ 별도 Python 서버 필요 |
| **성능 (동시 처리)** | ✅ Async 지원, 충분 | ✅ Event loop 기반 강점 |
| **Java 전환 용이성** | ⚠️ 비즈니스 로직 재작성 필요 | ⚠️ 비즈니스 로직 재작성 필요 |
| **양돈 데이터 분석** | ✅ 피그플랜 기존 분석 로직 재사용 | ❌ 재구현 필요 |
| **팀 역량** | 기존 팀이 Python 쓴다면 유리 | JS/TS 팀이라면 유리 |

**→ 권고: FastAPI 선택**
이유: AI 분석(질병 경보·최적 출하 예측) 기능이 Phase 2~3에 필수.
Python 생태계 없이는 별도 ML 서버 구축 비용 2배. 피그플랜 기존 분석 로직 재사용 가능.

### 3.2 Java 전환 시점 판단 기준

```
Phase 1 (FastAPI):  농장 수 ~300개, 동시 사용자 ~1,000명 이하
Phase 2 (FastAPI):  농장 수 ~1,000개, 동시 사용자 ~5,000명 이하

⚠️ 전환 트리거 조건 (셋 중 하나라도 해당 시):
  - API 응답 P95 > 500ms 지속 (2주 이상)
  - 월간 활성 농장 2,000개 초과
  - Enterprise 계약 SLA 99.9% 보장 요구

→ Java 전환은 "성능 문제 생기면"이 아니라
  "성능 문제가 예측될 때" 미리 준비 시작
```

### 3.3 PostgreSQL 설계 핵심 원칙

```
Schema-per-tenant (테넌트별 스키마 분리)
  ├── 장점: 농장별 데이터 완전 격리, 규정 준수 용이, 백업 단위 명확
  ├── 단점: 농장 수 늘어날수록 관리 복잡 (→ 자동화 필수)
  └── 결론: Phase 1~2는 Schema-per-tenant. Phase 3(수만 농장)에서 재검토.

핵심 테이블 구조 (논리적):
  Farm (농장) → Sow (모돈) → Event (이벤트: 교배/분만/이유)
                           → Health (건강 기록)
  Farm → FeedBin (사료빈) → FeedRecord (사료 기록)
  Farm → AnimalGroup (그룹) → GrowRecord (비육 기록)

TimescaleDB 사용 범위:
  IoT 센서 데이터 (환경 온도·습도, ESF 급이량) — 시계열 쿼리 최적화
  일반 농장 관리 데이터는 표준 PostgreSQL
```

## 4. Oracle MCP 분석 — 무엇을 봐야 하는가

> **핵심 메시지:** 27년 Oracle DB에서 "쓸 수 있는 것"과 "버려야 하는 것"을 먼저 구분해야 한다.
> 이걸 모르고 개발 시작하면 중간에 전부 뜯어고치는 사태가 발생한다.

### 4.1 Oracle MCP로 확인해야 할 항목

| 분석 항목 | 확인 내용 | 결과에 따른 결정 |
|---------|---------|--------------|
| **테이블 수·구조** | 총 테이블 수, 핵심 엔티티 파악 | PostgreSQL 설계 범위 결정 |
| **모돈 관련 테이블** | SOW, MATING, FARROWING, WEANING 등 | PigOS AI 핵심 스키마 재사용 여부 |
| **이벤트 로그 구조** | 이벤트 타입 종류, 기록 방식 | 48종 이벤트 지원 설계 기반 |
| **레거시 프로시저** | 커스텀 저장 프로시저·트리거 수 | 재구현 공수 추정 |
| **데이터 품질** | NULL 비율, 중복 레코드, 인코딩 문제 | 마이그레이션 정제 작업 범위 |
| **관계 무결성** | FK 관계, 고아 레코드 존재 여부 | PostgreSQL 전환 시 오류 위험 |
| **PSY/NPD 계산 로직** | 기존 KPI 계산 방식 | PigOS AI에서 동일하게 구현 가능한지 |

### 4.2 분석 결과에 따른 시나리오

```
시나리오 A: 구조가 재사용 가능 (이상적)
  Oracle 핵심 테이블 → PostgreSQL 직접 매핑
  소요 기간: 2주 분석 + 2주 마이그레이션 설계
  리스크: 낮음

시나리오 B: 일부 재사용, 일부 재설계
  핵심 엔티티(모돈·이벤트)만 재사용,
  국내 특화 로직은 제거하고 글로벌 표준으로 재설계
  소요 기간: 2주 분석 + 4~6주 설계
  리스크: 중간

시나리오 C: 구조 복잡도 높아 처음부터 설계
  PigOS AI = 완전 신규 설계. 국내 Oracle과 독립 운영
  소요 기간: 2주 분석 + 8주 신규 설계
  리스크: 높음 (일정 지연)
  → 이 시나리오가 나오면 회의에서 일정 재조정 필요
```

**→ 이번 회의 전에 Oracle MCP 분석 2주차 결과 공유 필수.**
어떤 시나리오인지 확정 못하면 개발 일정 전체가 불확실.

## 5. MVP 현실성 체크

> **핵심 메시지:** 8주 안에 "완벽한 제품"은 불가능하다.
> 그러나 "베트남 농장주 1명이 내일 쓸 수 있는 것"은 가능하다.
> 대표님께 이 두 가지를 명확히 구분해서 보고해야 한다.

### 5.1 8주 MVP — 현실적으로 가능한 것

> **기준:** 개발자 2~3명 기준. Oracle 분석 병행.

| 주차 | 작업 내용 | 결과물 |
|------|---------|-------|
| 1~2주 | Oracle MCP 분석 + PostgreSQL 스키마 설계 | ERD 초안, 테이블 정의서 |
| 3~4주 | FastAPI 기본 CRUD (모돈·이벤트·사용자) | API 엔드포인트 30개 |
| 5~6주 | React 웹 대시보드 기본 화면 + PSY/NPD | 웹에서 데이터 입력·조회 가능 |
| 7~8주 | Android 앱 오프라인 기본 기능 | 현장에서 교배/분만 기록 가능 |

**8주 후 가능한 것:**
- 모돈 교배·분만·이유 기록 (웹 + Android)
- PSY·NPD 기본 계산·표시
- 오프라인 입력 후 온라인 동기화 (기본)
- 한국어·영어·베트남어 UI

**8주 후 불가능한 것 (솔직하게):**
- Feed Mill 연동, Multi-Site, Genetics, ESG, ESF 연동
- iOS 앱, 벤치마킹, 고급 리포트

### 5.2 MVP로 무엇을 검증할 것인가

```
MVP의 목적 = "이게 팔릴까?" 가 아니라
             "이게 실제로 쓰일까?" 를 검증하는 것

검증 질문 3가지:
  1. 베트남 농장주가 앱을 스스로 설치하고 첫 데이터를 입력할 수 있는가?
  2. 인터넷 없는 환경에서 3일간 기록 후 동기화가 정상 작동하는가?
  3. PSY 수치를 보고 "이게 맞네"라고 반응하는가?

검증 방법:
  파일럿 농장 3~5개 모집 (한국계 베트남 진출 농장 우선)
  2주 실사용 후 인터뷰
  결과에 따라 Phase 2 방향 결정
```

### 5.3 대표님께 이렇게 보고한다

```
"8주 안에 완성 제품을 만드는 게 목표가 아닙니다.
 8주 안에 '실제로 쓰는 농장주 3명'을 확보하는 게 목표입니다.

 그 3명이 매일 데이터를 입력하고,
 'PSY가 올랐다'고 느끼면 —
 우리가 맞는 방향으로 가고 있다는 증거입니다.

 그게 현실적인 MVP입니다."
```

## 6. 회의 시나리오 — 대표님이 물어볼 것들

> 미리 답변을 준비해두면 회의 흐름을 주도할 수 있다.

### 예상 질문 & 준비된 답변

**Q1. "다 좋은데 돈은 언제 버나?"**
```
A: Phase 1의 목표는 수익이 아닙니다.
   활성 농장 300개 확보가 기준점입니다.
   그 시점에 유료 전환 방식을 결정합니다.
   보수적으로 300농장의 30% 전환 시 → 연 $160K~$450K ARR 진입 가능합니다.
   (구체적 수치는 7B 섹션 시나리오 참조)
```

**Q2. "PigKnows는 이미 글로벌 확장한다는데 우리가 이길 수 있나?"**
```
A: PigKnows가 가지지 못한 것이 3가지 있습니다.
   1. 아시아어 (한국어·베트남어·태국어) — 단기간에 추가 불가
   2. 무료 진입 — Valstone 구조상 무료 불가
   3. 오픈 API — 폐쇄형 수직통합 전략과 모순
   이 3가지 공백에서 우리가 먼저 자리를 잡으면 됩니다.
```

**Q3. "FastAPI로 시작했다가 Java로 바꾸면 처음부터 다시 짜야 하는 거 아닌가?"**
```
A: 비즈니스 로직 재작성은 필요합니다.
   그러나 DB 스키마, API 설계, 프론트엔드는 그대로 씁니다.
   Java 전환은 트래픽 문제가 예측될 때 시작합니다.
   그 시점이면 수익도 나고 있을 겁니다.
   지금 Java로 시작하면 개발 속도가 2~3배 느려집니다.
```

**Q4. "Oracle 분석이 생각보다 복잡하면 일정이 얼마나 밀리나?"**
```
A: 최악 시나리오(완전 신규 설계)는 8주가 추가됩니다.
   그러나 PigOS AI DB는 국내 레거시와 처음부터 독립 설계합니다.
   Oracle 분석은 "재사용 가능한 로직 찾기"가 목적이지,
   "반드시 쓰겠다"가 아닙니다.
   분석 결과와 무관하게 PigOS AI 개발은 병행 가능합니다.
```

**Q5. "베트남 농장주가 앱을 실제로 쓸 수 있을까? 너무 어렵지 않을까?"**
```
A: 이게 MVP에서 검증해야 할 핵심 질문입니다.
   피그플랜 앱 기준으로, 현장 작업자는 3개 화면만 씁니다:
   교배 기록 / 분만 기록 / 이유 기록
   이 3개 화면을 최대한 단순하게 만드는 것이 MVP UX 목표입니다.
```
