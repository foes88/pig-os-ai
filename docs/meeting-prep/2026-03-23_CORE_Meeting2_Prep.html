03
기술 스택 확인 및 진행 사항
"지금 빠르게"와 "나중에 버티게" 사이의 균형. 잘못 고르면 전부 다시 짜야 한다.
3.1 백엔드 선택: FastAPI vs Node.js
기준FastAPI (Python)Node.js (TypeScript)
개발 속도빠름 — Python 팀 역량 활용빠름 — JS 생태계
AI/ML 연동최강 — NumPy, pandas, 모델 직접 호출별도 Python 서버 필요
성능 (동시 처리)Async 지원, 충분Event loop 기반 강점
Java 전환 용이성비즈니스 로직 재작성 필요비즈니스 로직 재작성 필요
양돈 데이터 분석Python 데이터 분석 생태계 최강별도 Python 서버 필요
팀 역량기존 팀이 Python 쓴다면 유리JS/TS 팀이라면 유리
권고: FastAPI(Python) 선택. AI 분석(질병 경보/최적 출하 예측) 기능을 만들려면 Python이 필수. 다른 기술로 시작하면 나중에 Python을 따로 붙여야 해서 이중 작업이 된다.
3.2 왜 나중에 Java로 전환하는가?
한 줄 요약: Python은 빠르게 만들기 좋고, Java는 대규모로 버티기 좋다. 지금은 빠르게 만드는 게 먼저.
왜 Java인가?
  - 대규모 트래픽 처리에 강함 (수천 농장 동시 접속)
  - 엔터프라이즈 안정성 (SLA 99.9% 보장 가능)
  - 대형 고객(인테그레이터)이 요구하는 기술 스택

왜 지금은 안 하는가?
  - Java로 시작하면 개발 속도가 2~3배 느려짐
  - Phase 1~2 규모에서는 Python으로 충분

Python으로 충분한 구간:
  Phase 1: 농장 ~300개, 동시 사용자 ~1,000명
  Phase 2: 농장 ~1,000개, 동시 사용자 ~5,000명

Java 전환 트리거 (셋 중 하나라도 해당 시):
  - API 응답이 느려지기 시작할 때 (P95 > 500ms, 2주 이상)
  - 월간 활성 농장 2,000개 초과
  - 대형 고객이 SLA 99.9% 계약을 요구할 때

→ 전환 시점이 오면 이미 수익이 나고 있을 것.
  DB/API/프론트는 그대로 쓰고 비즈니스 로직만 재작성.
3.3 PostgreSQL 설계 핵심 원칙
Schema-per-tenant (테넌트별 스키마 분리)
  장점: 농장별 데이터 완전 격리, 규정 준수 용이, 백업 단위 명확
  단점: 농장 수 늘어날수록 관리 복잡 (→ 자동화 필수)
  결론: Phase 1~2는 Schema-per-tenant. Phase 3(수만 농장)에서 재검토.

핵심 테이블 구조 (논리적):
  Farm (농장) → Sow (모돈) → Event (이벤트: 교배/분만/이유)
                          → Health (건강 기록)
  Farm → FeedBin (사료빈) → FeedRecord (사료 기록)
  Farm → AnimalGroup (그룹) → GrowRecord (비육 기록)

TimescaleDB 사용 범위:
  IoT 센서 데이터 (환경 온도/습도, ESF 급이량) — 시계열 쿼리 최적화
  일반 농장 관리 데이터는 표준 PostgreSQL
Bottom Line FastAPI + PostgreSQL (Schema-per-tenant) + TimescaleDB(IoT). Java 전환은 성능 문제가 예측될 때 시작한다.
04
DB 설계 방향 — 완전 신규
기존 시스템과 독립적으로 글로벌 표준 기반 신규 설계로 진행한다.
Key Message
CORE는 기존 시스템(피그플랜 Oracle DB)과 완전히 독립된 신규 설계로 진행한다.
레거시에 얽매이지 않고, 글로벌 양돈 표준에 맞춰 처음부터 설계하는 것이 더 빠르고 깨끗하다.
4.1 신규 설계 원칙
원칙내용이유
완전 독립기존 DB 구조를 참고하지 않음레거시 제약 없이 글로벌 표준으로 설계 가능
Schema-per-tenant농장별 스키마 분리데이터 완전 격리, 규정 준수, 농장 단위 백업
5개 권역 동시 대응country_configs로 지역별 설정 관리단일 스키마로 EU/US/BR/SEA/CN 커버
모듈형 구조11개 모듈 독립 배포 가능Phase별 점진 확장에 유리
감사 추적모든 데이터 변경 → audit_log규정 준수 + 데이터 신뢰성
오프라인 동기화sync_queue + Last-Write-Wins인터넷 없는 농장 환경 대응
기존 시스템 분석 불필요. 완전 신규 설계로 확정. 레거시 의존성 없이 바로 개발 착수 가능.
Bottom Line CORE DB는 처음부터 새로 설계한다. 기존 시스템 분석 단계를 건너뛰고 바로 개발에 들어간다.

KPI 계산 공식 (핵심만)
PSY = SUM(이유두수) [12개월] / AVG(활성모돈수)
  // 위탁 자돈은 이유 모돈에 귀속, 연도 중간 입식은 pro-rata

NPD = SUM(이유~재교배 간격) [12개월]
  // 임신실패 재발정 = 전체 기간 NPD, 60일 초과 시 cap

FCR = SUM(사료소비 kg) / SUM(체중증가 kg)
  // 그룹 기반, 폐사두 사료 포함·체중 제외

분만율 = 분만 모돈수 / 교배 모돈수 × 100
  // 코호트: 110~150일 전 교배 건 대상, 폐사 제외

ROI = "FCR 0.1↓ = 두당 3,300원 절감" ← 실시간 금액 환산
  // Blue Ocean: 경쟁사 0/6 보유
