03
기술 스택 결정 사항
"지금 빠르게"와 "나중에 버티게" 사이의 균형. 잘못 고르면 6개월 후 전부 다시 짜야 한다.
3.1 백엔드 선택: FastAPI vs Node.js
기준FastAPI (Python)Node.js (TypeScript)
개발 속도빠름 — Python 팀 역량 활용빠름 — JS 생태계
AI/ML 연동최강 — NumPy, pandas, 모델 직접 호출별도 Python 서버 필요
성능 (동시 처리)Async 지원, 충분Event loop 기반 강점
Java 전환 용이성비즈니스 로직 재작성 필요비즈니스 로직 재작성 필요
양돈 데이터 분석Python 데이터 분석 생태계 최강별도 Python 서버 필요
팀 역량기존 팀이 Python 쓴다면 유리JS/TS 팀이라면 유리
권고: FastAPI 선택. AI 분석(질병 경보/최적 출하 예측) 기능이 Phase 2~3에 필수. Python 생태계 없이는 별도 ML 서버 구축 비용 2배.
3.2 Java 전환 시점 판단 기준
Phase 1 (FastAPI): 농장 수 ~300개, 동시 사용자 ~1,000명 이하
Phase 2 (FastAPI): 농장 수 ~1,000개, 동시 사용자 ~5,000명 이하

전환 트리거 조건 (셋 중 하나라도 해당 시):
  - API 응답 P95 > 500ms 지속 (2주 이상)
  - 월간 활성 농장 2,000개 초과
  - Enterprise 계약 SLA 99.9% 보장 요구

→ Java 전환은 "성능 문제 생기면"이 아니라
  "성능 문제가 예측될 때" 미리 준비 시작
3.3 PostgreSQL 설계 핵심 원칙
Schema-per-tenant (테넌트별 스키마 분리)
  장점: 농장별 데이터 완전 격리, 규정 준수 용이, 백업 단위 명확
  단점: 농장 수 늘어날수록 관리 복잡 (→ 자동화 필수)
  결론: Phase 1~2는 Schema-per-tenant. Phase 3(수만 농장)에서 재검토.

핵심 테이블 구조 (논리적):
  Farm (농장) → Sow (모돈) → Event (이벤트: 교배/분만/이유)
                          → Health (건강 기록)
  Farm → FeedBin (사료빈) → FeedRecord (사료 기록)
  Farm → AnimalGroup (그룹) → GrowRecord (비육 기록)

TimescaleDB 사용 범위:
  IoT 센서 데이터 (환경 온도/습도, ESF 급이량) — 시계열 쿼리 최적화
  일반 농장 관리 데이터는 표준 PostgreSQL
Bottom Line FastAPI + PostgreSQL (Schema-per-tenant) + TimescaleDB(IoT). Java 전환은 성능 문제가 예측될 때 시작한다.
04
Oracle MCP 분석
27년 Oracle DB에서 "쓸 수 있는 것"과 "버려야 하는 것"을 먼저 구분해야 한다.
Key Message
이걸 모르고 개발 시작하면 중간에 전부 뜯어고치는 사태가 발생한다.
4.1 Oracle MCP로 확인해야 할 항목
분석 항목확인 내용결과에 따른 결정
테이블 수/구조총 테이블 수, 핵심 엔티티 파악PostgreSQL 설계 범위 결정
모돈 관련 테이블SOW, MATING, FARROWING, WEANING 등CORE 핵심 스키마 재사용 여부
이벤트 로그 구조이벤트 타입 종류, 기록 방식48종 이벤트 지원 설계 기반
레거시 프로시저커스텀 저장 프로시저/트리거 수재구현 공수 추정
데이터 품질NULL 비율, 중복 레코드, 인코딩 문제마이그레이션 정제 작업 범위
관계 무결성FK 관계, 고아 레코드 존재 여부PostgreSQL 전환 시 오류 위험
PSY/NPD 계산 로직기존 KPI 계산 방식CORE에서 동일하게 구현 가능한지
4.2 분석 결과에 따른 시나리오
SCENARIO A
이상적

구조가 재사용 가능

Oracle 핵심 테이블 → PostgreSQL 직접 매핑

소요: 2주 분석 + 2주 마이그레이션 설계

리스크: 낮음

SCENARIO B
현실적

일부 재사용, 일부 재설계

핵심 엔티티(모돈/이벤트)만 재사용, 국내 특화 로직 제거 후 글로벌 표준 재설계

소요: 2주 분석 + 4~6주 설계

리스크: 중간

SCENARIO C
최악

처음부터 설계

CORE = 완전 신규 설계. 국내 Oracle과 독립 운영

소요: 2주 분석 + 8주 신규 설계

리스크: 높음 (일정 지연)

이번 회의 전에 Oracle MCP 분석 2주차 결과 공유 필수. 어떤 시나리오인지 확정 못하면 개발 일정 전체가 불확실.
Bottom Line Oracle 분석 결과가 전체 일정을 결정한다. 시나리오 확정이 이번 회의 최우선 과제.
05
MVP 현실성 체크
8주 안에 "완벽한 제품"은 불가능. "베트남 농장주 1명이 내일 쓸 수 있는 것"은 가능.
Key Message
이 두 가지를 명확히 구분해야 한다.
"가능한 것"과 "불가능한 것"을 미리 선 그어야 기대치를 관리할 수 있다.

5.2 MVP로 무엇을 검증할 것인가
MVP의 목적 = "이게 팔릴까?"가 아니라 "이게 실제로 쓰일까?"를 검증하는 것
검증 질문 3가지
#질문
1베트남 농장주가 앱을 스스로 설치하고 첫 데이터를 입력할 수 있는가?
2인터넷 없는 환경에서 3일간 기록 후 동기화가 정상 작동하는가?
3PSY 수치를 보고 "이게 맞네"라고 반응하는가?
검증 방법
  • 파일럿 농장 3~5개 모집 (한국계 베트남 진출 농장 우선)
  • 2주 실사용 후 인터뷰
  • 결과에 따라 Phase 2 방향 결정
5.3 MVP 핵심 메시지
"8주 안에 완성 제품을 만드는 게 목표가 아닙니다.
8주 안에 '실제로 쓰는 농장주 3명'을 확보하는 게 목표입니다.

그 3명이 매일 데이터를 입력하고,
'PSY가 올랐다'고 느끼면 —
우리가 맞는 방향으로 가고 있다는 증거입니다.

그게 현실적인 MVP입니다."
Bottom Line 8주 MVP = 베트남 농장주 3명이 실제로 쓰는 것. 완성 제품이 아니라 검증이 목적이다.
06
회의 시나리오 — 예상 질문 & 준비된 답변
미리 답변을 준비해두면 회의 흐름을 주도할 수 있다.
Q1"다 좋은데 돈은 언제 버나?"
Phase 1의 목표는 수익이 아닙니다.
활성 농장 300개 확보가 기준점입니다.
그 시점에 유료 전환 방식을 결정합니다.
보수적으로 300농장의 30% 전환 시 → 연 $160K~$450K ARR 진입 가능합니다.
Q2"PigKnows는 이미 글로벌 확장한다는데 우리가 이길 수 있나?"
PigKnows가 가지지 못한 것이 3가지 있습니다.
1. 아시아어 (한국어/베트남어/태국어) — 단기간에 추가 불가
2. 무료 진입 — Valstone 구조상 무료 불가
3. 오픈 API — 폐쇄형 수직통합 전략과 모순
이 3가지 공백에서 우리가 먼저 자리를 잡으면 됩니다.
Q3"FastAPI로 시작했다가 Java로 바꾸면 처음부터 다시 짜야 하는 거 아닌가?"
비즈니스 로직 재작성은 필요합니다.
그러나 DB 스키마, API 설계, 프론트엔드는 그대로 씁니다.
Java 전환은 트래픽 문제가 예측될 때 시작합니다.
그 시점이면 수익도 나고 있을 겁니다.
지금 Java로 시작하면 개발 속도가 2~3배 느려집니다.
Q4"Oracle 분석이 생각보다 복잡하면 일정이 얼마나 밀리나?"
최악 시나리오(완전 신규 설계)는 8주가 추가됩니다.
그러나 CORE DB는 국내 레거시와 처음부터 독립 설계합니다.
Oracle 분석은 "재사용 가능한 로직 찾기"가 목적이지, "반드시 쓰겠다"가 아닙니다.
분석 결과와 무관하게 CORE 개발은 병행 가능합니다.
Q5"베트남 농장주가 앱을 실제로 쓸 수 있을까? 너무 어렵지 않을까?"
이게 MVP에서 검증해야 할 핵심 질문입니다.
피그플랜 앱 기준으로, 현장 작업자는 3개 화면만 씁니다:
교배 기록 / 분만 기록 / 이유 기록
이 3개 화면을 최대한 단순하게 만드는 것이 MVP UX 목표입니다.

KPI 계산 공식 (핵심만)
PSY = SUM(이유두수) [12개월] / AVG(활성모돈수)
  // 위탁 자돈은 이유 모돈에 귀속, 연도 중간 입식은 pro-rata

NPD = SUM(이유~재교배 간격) [12개월]
  // 임신실패 재발정 = 전체 기간 NPD, 60일 초과 시 cap

FCR = SUM(사료소비 kg) / SUM(체중증가 kg)
  // 그룹 기반, 폐사두 사료 포함·체중 제외

분만율 = 분만 모돈수 / 교배 모돈수 × 100
  // 코호트: 110~150일 전 교배 건 대상, 폐사 제외

ROI = "FCR 0.1↓ = 두당 3,300원 절감" ← 실시간 금액 환산
  // Blue Ocean: 경쟁사 0/6 보유
