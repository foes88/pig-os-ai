#7. Schema-per-tenant 조사

결론: PigPlanCORE에 최적. 단, 확장 경로를 미리 잡아둬야 한다.

방식격리 수준확장 한계복잡도적합성
Database-per-tenant최대~10개높음과잉
Schema-per-tenant강력수백~수천 개중간최적
Row-level Security (RLS)약함수백만낮음시기상조

장점

  • 데이터 완전 격리 — A농장 쿼리가 B농장 데이터에 접근 불가 (구조적으로)
  • 백업/복원 농장 단위 — 특정 농장만 백업, 복원, 삭제 가능
  • 규정 준수 용이 — 국가별 데이터 보관 요구사항 대응
  • 농장별 커스터마이징 — 스키마 단위로 농장별 설정 가능
  • 마이그레이션 단순 — 농장 단위로 독립적 스키마 마이그레이션

단점/리스크

  • 스키마 수 증가 시 성능 저하 — PostgreSQL 카탈로그 크기 증가. 수천 개 이상에서 느려질 수 있음
  • 커넥션 풀링 복잡 — PgBouncer Transaction Pooling 필수. 매 트랜잭션마다 search_path 설정 필요
  • 스키마 마이그레이션 반복 — 테이블 변경 시 모든 스키마에 반복 적용해야 함

농장 수 늘어나면?

농장 수상태대응
~500개문제 없음기본 PostgreSQL로 충분
500~2,000개양호PgBouncer 필수. 모니터링 시작
2,000~5,000개주의카탈로그 최적화 + 커넥션 관리 강화
5,000개+전환 검토Citus 12 Schema-based Sharding으로 수평 확장
확장 경로: Schema-per-tenant → Citus 12 Schema-based Sharding. Citus는 PostgreSQL 확장이라 기존 스키마 구조를 유지하면서 분산 처리 가능. 마이그레이션 도구(pgcopydb) 지원.
출처: Crunchy Data Blog, Citus 12 공식 문서, debugg.ai Multi-tenancy Playbook 2025
